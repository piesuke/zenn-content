こんにちは。[株式会社 Sally](https://sally-inc.jp/) エンジニアの [@piesuke](https://x.com/piesuke27)です。
私たちは、マーダーミステリーを遊ぶことが出来るアプリ「ウズ」と、マーダーミステリーを制作してウズ上で遊べることが出来るアプリ「ウズスタジオ」を開発しています。
最近良かったマーダーミステリーは「[ザ・シークレットワン・ショータイム！](https://mdms.jp/scenarios/10432)」です。

Claude Codeを活用するようになり、開発速度が劇的に向上しました。大いに助かっているのですが、人間とは強欲なもので、一回一回のセッションをもっと効率的にこなしたいと常に思います。また、AIに実装をやってもらうだけでは人間は特に賢くなっておらず、無力感も覚えます。このような課題を感じたので、セッションが終わった際にAIに振り返りを行ってもらい、人間もAIもセッションごとに賢くなってもらおうと思い立ちました。

## skill-creatorを使って振り返りスキルを作成する

スキルはClaude Codeから公式に提供されている `skill-creator` を使って作成します。`/skill-creator` とコマンドを打つと「どんなスキルを作りたいのか」と、スキルを作成するために必要な情報が対話式に表示されるので適宜入力します。私は今回、

```markdown
セッションが完了した後、そのセッションがより良くなるようにclaudeの設定を修正してくれて、かつそのセッションをこなすために必要だったCSの知識や開発の原則を教えてくれるスキル
```

と入力しました。セッション名は `session-retro` という名前にしました。フォルダ構成は最小限で以下のような構成です。

```
session-retro/
├── SKILL.md
└── references/
    └── cs-principles.md
```

## session-retroの設計思想

このスキルは3つのPhaseで構成されています。

### Phase 1: セッション分析

会話履歴から、実行したタスクの要約・遭遇した問題・使用した技術パターン・繰り返し出現したパターンを抽出します。

### Phase 2: 設定改善

セッションの知見に基づいて、以下の設定ファイルへの改善をチェックリスト形式で提案します。

- **CLAUDE.md** — コーディング規約の追加、よく使うコマンドの追記など
- **Auto Memory** — デバッグパターンやプロジェクト固有の知見の記録
- **settings.json** — セッション中に毎回許可した操作の自動許可化など
- **Skills / Commands** — 繰り返し行った操作のスキル化提案

提案はあくまでチェックリストとして表示され、ユーザーがOKした項目のみ適用されます。

### Phase 3: CS知識・開発原則の解説

セッションに関連したCS知識や開発原則を、`references/cs-principles.md` に定義された原則カタログ（SOLID原則、デザインパターン、型システムなど12分野）から最大5つピックアップし、セッションの具体的な文脈と紐づけて解説します。

SKILL.mdの全文は以下のようになっております。
:::details SKILL.md全文
```markdown
---
name: session-retro
description: セッション完了後の振り返りスキル。セッションの内容を分析し、(1) Claude の設定ファイル（CLAUDE.md、auto memory、settings.json、.claude ディレクトリ）を改善提案・修正、(2) セッションで関連したCS知識や開発原則をターミナルに教育的に解説する。セッション終了時や「振り返り」「retro」「レトロ」と言われた時に使用する。
---

# session-retro

セッション完了後に振り返りを行い、設定の改善とCS知識の教育を提供する。

## 処理フロー

### Phase 1: セッション分析

会話履歴から以下を抽出する：

1. **実行したタスクの要約**: 何をしたか、どのファイルを変更したか
2. **遭遇した問題・つまずき**: エラー、非効率な手順、情報不足
3. **使用した技術・パターン**: 言語、フレームワーク、設計パターン
4. **繰り返し出現したパターン**: 何度も参照した情報、繰り返した手順

### Phase 2: 設定改善

以下の設定ファイルを読み、セッションの知見に基づいて改善提案を行う。

#### 対象ファイル

1. **CLAUDE.md** (プロジェクト / ディレクトリ単位)
2. **Auto Memory** (`~/.claude/projects/` 配下の `memory/` ディレクトリ)
3. **settings.json** (`~/.claude/settings.json` / プロジェクト配下)
4. **Skills / Commands** (`.claude/skills/` および `.claude/commands/`)

#### 改善提案のルール

- **変更は必ずユーザーに確認してから適用する**
- 既存の設定と矛盾する変更は提案しない
- 一時的・セッション固有の情報は設定に入れない
- 提案は箇条書きで簡潔に。各提案に「なぜ必要か」の理由を1行で添える

ユーザーがチェックを入れた項目のみ適用する。

### Phase 3: CS知識・開発原則の解説

セッションで関連したCS知識や開発原則をターミナルに表示する。

1. references/cs-principles.md を読み、セッションに該当する原則を特定
2. 該当する原則について、概要・セッションとの関連・深掘り・参考文献の構造で解説する

#### 解説のルール

- セッションに直接関連する原則のみ（最大5つ）
- 抽象的な説明ではなく、セッションの具体的な文脈と紐づける
- 初学者にもわかるように、専門用語は初出時に簡潔に説明する
- 「なぜそうするのか」の理由を重視する

## 出力順序

1. セッション要約（3-5行）
2. 設定改善提案（Phase 2）
3. CS知識・開発原則の解説（Phase 3）
```
:::

## 実際の出力例

これを毎回セッション終わりに `/session-retro` と呼び出すだけで以下のようなフィードバックを送ってくれます。以下は例として、`UserProfile.tsx` にて `OldButton` を `NewButton` という新しいコンポーネントに置き換えたセッションのフィードバックです。

```
⏺ セッション要約
  UserProfile.tsx にて、旧コンポーネント OldButton を新しい NewButton に置き換えた。
  インポート文、型注釈、プロップ名（onClick → onPress）を変更。
  リンターは環境依存のエラーで実行できなかったが、変更自体はシンプルで問題なし。

⏺ 設定改善提案

  Memory

  - memory/MEMORY.md にリンターの実行方法を記録
    — 理由: npx eslint が worktree 環境で失敗した。正しい実行方法（npm run lint 経由）
    を記録しておくと次回スムーズ
  - memory/component-migration.md を新規作成し、OldButton → NewButton の prop 差分を記録
    — 理由: 同様の置き換えが他ファイルでも発生する可能性がある

  settings.json

  - permissions に Bash(npm run lint *) を追加
    — 理由: リンターを npm run 経由で実行する場面が多く、都度許可が必要になるため

  CLAUDE.md / Skills

  特になし。

  ---
  CS知識・開発原則の解説

  ---
  Liskov Substitution Principle（リスコフの置換原則）

  概要: 親の型が使われている箇所で、子の型に置き換えてもプログラムの正しさが
  保たれなければならないという原則。SOLID 原則の「L」。

  今回のセッションとの関連:
  OldButton を NewButton に置き換える際、両者の props は似ているが完全には
  一致しなかった（onClick → onPress）。共通のインターフェースを持っていれば、
  呼び出し側の変更なしに置き換えられたはずである。

  深掘り:
  UIコンポーネントにおけるこの原則は、「あるコンポーネントを別のコンポーネントに
  差し替えたとき、呼び出し側をどれだけ変更する必要があるか」で考えると実感しやすい。
  同カテゴリのコンポーネントが共通のインターフェース契約を持てば移行コストは
  最小化できる。

  参考: "Clean Architecture" by Robert C. Martin、SOLID 原則
```

設定改善提案に対して、ユーザー側がOKすればClaude Code側で改善を行ってくれます。
また、`CS知識・開発原則の解説` パートでは、原則名や用語名と合わせて教えてくれるので、気になった原則に対しての自分での深掘りも出来やすいかなと思います。また、自分用のメモファイルを用意し、そこにフィードバックを受けた原則を溜めることで、後々勉強するときにも繋がるかなと思います。

## まとめ

Claude Codeにセッションの自己改善を行ってもらい、人間とAIを同時に賢くしていきましょう。
